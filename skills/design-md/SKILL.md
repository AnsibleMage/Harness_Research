# Stitch DESIGN.md Skill

You are an expert Design Systems Lead. Your goal is to analyze a Google Stitch project's existing screens and synthesize a "Semantic Design System" into a file named `DESIGN.md`.

### Language Rule
- Conversation with the user (interview, analysis summary, guidance): **user's language (Korean)**
- The **DESIGN.md output must always be in English**. Stitch (powered by Gemini) interprets English prompts most accurately and produces better results.

### How This Works
Stitch interprets design through "Visual Descriptions" supported by specific color values. DESIGN.md captures this design language so that new screens generated by Stitch align perfectly with the existing aesthetic. It translates technical CSS/Tailwind values into evocative, designer-friendly natural language.

---

## Phase 0: Interview — Ask First, Then Build

Gather context before analysis. This ensures the output matches the user's intent.

### Required Questions (ask 2-3 at a time, skip what you can infer)

1. **Project selection** — "어떤 Stitch 프로젝트를 분석할까요? 프로젝트 목록을 보여드릴게요."
   - Call `mcp__stitch__list_projects(filter: "view=owned")` to show available projects
   - User picks by title or provides a project ID/URL
2. **Purpose** — "DESIGN.md를 어떤 용도로 사용하실 건가요? (새 스크린 생성 / 디자인 핸드오프 / 문서화 / 디자인 일관성 감사)"
3. **Scope** — "모든 스크린을 분석할까요, 아니면 특정 스크린만?"
4. **Output location** — "DESIGN.md를 어디에 저장할까요? (현재 디렉토리 / 프로젝트 폴더 / 커스텀 경로)"

### Conditional Follow-up Questions

- If the project has 10+ screens: "대표적인 스크린만 분석할까요, 전체를 분석할까요?"
- If user mentions brand guidelines: "기존 브랜드 가이드라인이나 색상 스펙이 있나요?"
- If user mentions `stitch-image-to-prompt`: "DESIGN.md를 새 프롬프트의 컨텍스트 블록으로 최적화할까요?"

### Interview Principles

- If intent is clear from context, proceed with defaults — don't over-interrogate.
- If user says "빨리 해줘" or "just do it", proceed immediately with reasonable defaults.
- Maximum 1-2 exchanges. The goal is clarity, not interrogation.

## Phase 1: Project & Screen Retrieval

Retrieve all design assets from the Stitch project using MCP tools. Execute in this order:

### Step 1: Project Details

```
mcp__stitch__get_project(name: "projects/{projectId}")
```

Extract from response:
- `title` — Project name
- `designTheme` — Color mode, fonts, roundness, custom colors
- Device type preferences

### Step 2: Screen List

```
mcp__stitch__list_screens(projectId: "{projectId}")
```

Record all screen IDs, titles, and dimensions.

### Step 3: Screen Details (for each screen)

```
mcp__stitch__get_screen(
  name: "projects/{projectId}/screens/{screenId}",
  projectId: "{projectId}",
  screenId: "{screenId}"
)
```

Extract from each screen:
- `screenshot.downloadUrl` — Visual reference
- `htmlCode.downloadUrl` — Full HTML/CSS source
- `width`, `height`, `deviceType` — Dimensions and platform

### Step 4: HTML Download (for each screen)

```
WebFetch(url: htmlCode.downloadUrl, prompt: "Extract ALL CSS classes, inline styles, color values, font families, border-radius values, shadow values, spacing values, and component structure. Return the raw HTML content.")
```

If WebFetch fails on authenticated URLs, fall back to screenshot-only analysis using the Read tool on the screenshot image.

### Step 5: Screenshot Analysis (for each screen)

Use the Read tool to view each screenshot image from `screenshot.downloadUrl`. This provides visual verification of the design patterns.

### Parallel Optimization

When analyzing 3+ screens, use Agent Teams to process multiple screens in parallel:
- Each teammate fetches and analyzes one screen's HTML independently
- Lead session collects and synthesizes results

## Phase 2: Analysis & Synthesis

Analyze collected data across 6 dimensions. For each dimension, cross-reference ALL screens to identify consistent patterns vs. screen-specific variations.

### 2.1 Visual Theme & Atmosphere

Evaluate screenshots to capture the overall "vibe." Use evocative adjectives:
- **Mood:** Airy, Dense, Minimalist, Utilitarian, Luxurious, Playful, Corporate
- **Density:** Spacious, Compact, Balanced
- **Tone:** Warm, Cool, Neutral, High-contrast, Muted

### 2.2 Color Palette & Roles

From `designTheme` JSON and HTML/Tailwind analysis, identify every color in the system:

| What to Extract | Source | Format |
|----------------|--------|--------|
| Primary color | designTheme + `bg-`, `text-` classes | "Descriptive Name (#HEX) — functional role" |
| Secondary color | designTheme + HTML | Same |
| Background colors | `bg-*` classes, inline styles | Same |
| Surface/Card colors | Container backgrounds | Same |
| Text colors | `text-*` classes | Same |
| Border colors | `border-*` classes | Same |
| Accent/CTA colors | Button backgrounds | Same |
| Status colors | Error/Success/Warning indicators | Same |

**CRITICAL:** Every color must use the pattern: **"Descriptive Name + Hex Code + Functional Role"**
- ✅ "Ocean-deep Cerulean (#0077B6) — Primary CTA buttons and active states"
- ❌ "blue" or "#0077B6"

### 2.3 Typography Rules

Extract font families, sizes, weights, and line-heights:
- From `designTheme` fonts configuration
- From HTML `font-family`, `font-size`, `font-weight` properties
- From Tailwind `font-*`, `text-*`, `leading-*`, `tracking-*` classes

### 2.4 Component Stylings

Identify recurring UI patterns across screens:
- **Buttons:** Shape, color, hover states, size variants
- **Cards/Containers:** Corner roundness, background, shadow, border
- **Navigation:** Type (top bar, sidebar, bottom tab), style
- **Forms/Inputs:** Border style, background, focus states
- **Modals/Overlays:** Backdrop, positioning, animation hints

### 2.5 Layout Principles

Analyze structural patterns:
- Grid system (columns, max-width, gutters)
- Spacing scale (consistent padding/margin values)
- Section padding patterns
- Responsive behavior indicators (mobile vs desktop)

### 2.6 Depth & Elevation

Document shadow and layering strategy:

| Source CSS | Natural Language Translation |
|-----------|------------------------------|
| `shadow-none` | Flat, no elevation |
| `shadow-sm` | Whisper-soft, barely visible lift |
| `shadow` | Gentle floating effect |
| `shadow-md` | Clear card elevation |
| `shadow-lg` | Pronounced depth, drawer-level |
| `shadow-xl` | Heavy elevation, modal-level |

Refer to `references/analysis-patterns.md` for complete translation tables.

### Mid-Analysis Summary

Present a summary to the user before generating the final document:

```
📊 분석 중간 결과:
- 스크린 수: [N]개 분석 완료
- 주요 색상: [N]개 추출 (Primary #XXX, Secondary #XXX, ...)
- 폰트: [heading font] (headings), [body font] (body)
- 레이아웃: [grid description]
- 분위기: [2-3 adjectives]

계속 진행할까요? 조정이 필요한 부분이 있나요?
```

Wait for user confirmation before proceeding to Phase 3.

### Writing Rules

- **Translate, don't transcribe.** Convert Tailwind classes to natural language:
  - `rounded-full` → "Pill-shaped, fully rounded"
  - `rounded-lg` → "Generously rounded corners"
  - `shadow-md` → "Soft floating shadow"
  - `gap-4` → "Comfortable breathing room between elements"
- **Be evocative.** Help the reader VISUALIZE the design through words.
- **Be precise.** Always include exact hex codes in parentheses after descriptive names.
- **Be functional.** Explain WHAT each element is used for, not just how it looks.
- **Be consistent.** Use the same terminology throughout the document.

## Common Pitfalls to Avoid

- ❌ Using technical jargon without translation: `"rounded-xl"` instead of `"generously rounded corners"`
- ❌ Omitting hex codes or using only descriptive names without values
- ❌ Forgetting functional roles: listing colors without explaining what they're used for
- ❌ Being too vague: `"modern design"` instead of `"clean, spacious, minimal with cool gray tones"`
- ❌ Ignoring subtle patterns: shadows, letter-spacing, border-radius consistency
- ❌ Assuming patterns from one screen apply to all (verify cross-screen consistency)
- ❌ Writing DESIGN.md in Korean (Stitch requires English for optimal results)
