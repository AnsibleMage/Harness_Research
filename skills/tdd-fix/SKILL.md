# TDD-Fix: 유지보수 최소 수정 스킬

> **철학**: 전체를 고치지 않는다. 안 되는 부분만 찾아서 그 부분만 고친다.

## 핵심 원칙: Surgical TDD

```
┌──────────────────────────────────────────────────────────┐
│  🔴 RED    — 뭐가 안 되는지 테스트시트로 정의한다         │
│  🟢 GREEN  — 테스트를 통과시키는 최소한의 코드만 고친다    │
│  ✅ VERIFY — 고친 것만 통과하는지, 다른 건 안 깨졌는지     │
│                                                          │
│  ⚠️ 절대 금지:                                            │
│  - 주변 코드 "개선"                                       │
│  - "이왕 하는 김에" 리팩토링                               │
│  - 동작하는 코드 구조 변경                                 │
│  - 요청받지 않은 파일 수정                                 │
└──────────────────────────────────────────────────────────┘
```

---

## Phase 0: INTAKE (접수)

사용자로부터 다음을 파악한다:

| 필수 정보 | 확인 방법 | 예시 |
|----------|----------|------|
| **증상** | 사용자 설명 | "목록이 안 나와", "버튼 클릭해도 반응 없어" |
| **대상 파일** | 사용자 지목 또는 탐색 | `board/list.jsp`, `BoardController.java` |
| **수정 범위** | 마크업 / Java / 둘 다 | "화면만 수정", "로직도 좀" |

파악되지 않은 항목은 **반드시 사용자에게 질문**한다. 추측하지 않는다.

---

## Phase 1: LOCATE (탐색) — 수정 범위 특정

### 1-1. 대상 파일 읽기

```
Read → 사용자가 지목한 파일 전체 또는 관련 영역
Explore → 파일 미지정 시 증상 기반 탐색
```

### 1-2. 문제 지점 격리

**마크업 (JSP/HTML/CSS):**
- 해당 섹션/블록의 시작~끝 라인 특정
- 관련 CSS 클래스/ID 식별
- include 되는 공통 파일 확인 (header, footer, common.jsp)

**Java (Controller/Service/DAO):**
- 문제 메서드 1개 특정 (메서드 단위로 격리)
- 호출 체인 확인: Controller → Service → ServiceImpl → DAO → SQL(mapper.xml)
- **수정할 메서드만** 집중, 나머지는 읽기만

### 1-3. 수정 범위 선언 (필수)

탐색 완료 후 반드시 아래 형식으로 범위를 선언한다:

```
📍 수정 범위 선언
─────────────────
파일: board/list.jsp
영역: 45~62번 라인 (목록 테이블 영역)
원인: <c:forEach> 변수명 불일치
수정 방식: 마크업 수정 (1곳)
영향 범위: 해당 테이블만 (다른 페이지 무관)
```

⚠️ **범위 선언 없이 수정 시작 금지.**

## Phase 2: RED (테스트시트 작성)

### TDD에서 "테스트"란

유지보수에서 테스트 = **"고친 후 이렇게 되어야 한다"는 기대 결과 명세**.
JUnit만이 테스트가 아니다. 수정 유형에 맞는 검증 수단을 선택한다.

### 2-1. 테스트시트 작성

수정 전에 반드시 아래 형식의 테스트시트를 작성한다:

```
📋 테스트시트 — [파일명] [증상 요약]
───────────────────────────────────
#  | 검증 항목          | 기대 결과              | 현재 상태     | 수정 후
───────────────────────────────────
T1 | 목록 데이터 출력    | 10건 테이블 표시        | 빈 화면       | □
T2 | 페이징 링크 동작    | 2페이지 클릭 시 이동    | 정상 (유지)   | □
T3 | 검색 기능          | 키워드 입력 시 필터링    | 정상 (유지)   | □
```

### 2-2. 테스트 유형별 가이드

| 수정 유형 | 테스트 방법 | 검증 포인트 |
|----------|-----------|-----------|
| **JSP 마크업** | 코드 리뷰 (구조/문법 확인) | EL 표현식, JSTL 태그, 변수명 일치, HTML 구조 |
| **CSS 스타일** | 속성값 확인 | 셀렉터 정확성, 속성 충돌, 우선순위 |
| **JavaScript** | 로직 트레이스 | 이벤트 바인딩, DOM 셀렉터, Ajax URL/파라미터 |
| **Java 메서드** | 입출력 트레이스 | 파라미터 매핑, 리턴 타입, null 처리 |
| **SQL (mapper.xml)** | 쿼리 로직 확인 | 파라미터 바인딩 (#{} vs ${}), WHERE 조건, 컬럼명 |

### 2-3. 회귀 방지 항목 (필수)

테스트시트에 **반드시 "정상 (유지)" 항목을 1개 이상** 포함한다.
→ 수정하지 않는 기존 기능이 깨지지 않았음을 확인하는 용도.

## Phase 3: GREEN (최소 수정)

### 3-1. 최소 수정 원칙

```
┌─ 수정 판단 기준 ─────────────────────────┐
│                                          │
│  "이 변경이 없으면 테스트가 실패하는가?"    │
│                                          │
│  YES → 수정한다                           │
│  NO  → 수정하지 않는다                    │
│                                          │
└──────────────────────────────────────────┘
```

### 3-2. 수정 유형별 규칙

#### 마크업 수정 (JSP/HTML/CSS) — 주력

| 규칙 | 설명 |
|------|------|
| **정확한 블록만 교체** | Edit tool의 old_string을 해당 블록만 매칭 |
| **주변 들여쓰기 보존** | 기존 파일의 인덴트 스타일 유지 (탭 vs 스페이스) |
| **기존 클래스명 유지** | 새 클래스 추가는 가능, 기존 클래스 제거/변경은 최소화 |
| **include 파일 미수정** | 공통 파일(header.jsp 등)은 영향 범위가 넓어 원칙적 미수정 |
| **EL/JSTL 문법 준수** | `${vo.boardId}` 형식, `<c:if>`, `<c:forEach>` 정확한 문법 |

#### Java 수정 — 보조

| 규칙 | 설명 |
|------|------|
| **메서드 단위 수정** | 메서드 시그니처 변경 최소화, 내부 로직만 수정 |
| **새 메서드 추가 지양** | 기존 메서드 내에서 해결 우선 |
| **import 추가 시 기록** | 새 import가 필요하면 테스트시트에 별도 기록 |
| **예외 처리 패턴 유지** | 기존 try-catch 구조 따름, 새 예외 패턴 도입 금지 |

#### eGovFrame 특수 규칙

| 대상 | 주의사항 |
|------|---------|
| **Controller** | `@RequestMapping` URL 변경 금지 (화면 연결 깨짐) |
| **Service/ServiceImpl** | 인터페이스-구현체 쌍 일치 확인 |
| **DAO (mapper.xml)** | namespace, id 변경 금지. SQL 내부만 수정 |
| **VO/DTO** | 필드 추가 시 getter/setter 쌍 필수 |
| **globals.properties** | 수정 시 전체 영향. 원칙적 미수정 |
| **context-*.xml** | Spring 설정 파일 수정 금지 (재기동 필요 + 사이드이펙트) |

### 3-3. 변경 추적

수정할 때마다 아래 형식으로 기록:

```
✏️ 변경 #1
파일: board/list.jsp:48
before: <c:forEach var="result" items="${resultList}">
after:  <c:forEach var="result" items="${list}">
이유: Controller에서 model.addAttribute("list", ...) 으로 전달
```

## Phase 4: VERIFY (패스 확인)

### 4-1. 테스트시트 패스 체크

테스트시트의 모든 항목을 순회하며 체크:

```
📋 테스트시트 — 최종 결과
───────────────────────────────────
#  | 검증 항목          | 기대 결과              | 현재 상태     | 수정 후
───────────────────────────────────
T1 | 목록 데이터 출력    | 10건 테이블 표시        | 빈 화면       | ✅ PASS
T2 | 페이징 링크 동작    | 2페이지 클릭 시 이동    | 정상 (유지)   | ✅ PASS
T3 | 검색 기능          | 키워드 입력 시 필터링    | 정상 (유지)   | ✅ PASS
```

### 4-2. 범위 초과 셀프 체크 (필수)

```
🔍 범위 초과 점검
─────────────────
✅ 선언한 수정 범위만 변경했는가?
✅ 요청받지 않은 파일을 수정하지 않았는가?
✅ "이왕 하는 김에" 추가 수정을 하지 않았는가?
✅ 동작하던 기능이 유지되는가?
✅ 공통 파일(header, common, config)을 건드리지 않았는가?
```

### 4-3. 변경 요약 리포트

```
📊 TDD-Fix 완료 리포트
═══════════════════════
증상: [원래 문제]
원인: [근본 원인 1줄]
수정: [변경 파일 수] 파일, [변경 라인 수] 라인
테스트: [통과 수]/[전체 수] PASS

변경 파일 목록:
  ✏️ board/list.jsp:48 — forEach 변수명 수정

회귀 확인:
  ✅ 페이징 정상
  ✅ 검색 정상
```

## Anti-Patterns (절대 하지 않는 것)

| # | Anti-Pattern | 왜 안 되는가 |
|---|-------------|-------------|
| 1 | **전체 파일 재작성** | 유지보수는 부분 수정이다. 전체를 바꾸면 검증 불가 |
| 2 | **코드 스타일 통일** | 동작하는 레거시 코드의 스타일은 건드리지 않는다 |
| 3 | **Deprecated API 교체** | 요청받지 않은 기술 부채 해소는 별도 작업이다 |
| 4 | **주석 추가/정리** | 수정 대상이 아닌 주석은 원본 유지 |
| 5 | **import 정리** | 사용하지 않는 import가 있어도 제거하지 않는다 |
| 6 | **변수명 개선** | 기존 변수명이 나빠도 인터페이스가 바뀌면 위험 |
| 7 | **공통 파일 수정** | header.jsp, common.css 등은 전체 화면에 영향 |
| 8 | **Spring 설정 변경** | context-*.xml, web.xml 수정은 별도 의사결정 필요 |

## 수정 유형별 빠른 참조

### 마크업 (JSP/HTML) — 가장 빈번

```
1. Read → 해당 JSP 파일
2. 문제 블록 라인 특정
3. 테스트시트 작성 (EL 표현식, 태그 구조, 변수명)
4. Edit → old_string 정확 매칭 → new_string 최소 변경
5. 변경 전후 diff 확인
```

### CSS 수정

```
1. Read → JSP에서 class/id 확인
2. Grep → 해당 셀렉터가 정의된 CSS 파일 찾기
3. 테스트시트 작성 (기대 스타일 속성)
4. Edit → 해당 속성만 수정 (블록 전체 교체 금지)
5. 같은 셀렉터 다른 파일 여부 확인 (중복 정의 주의)
```

### JavaScript 수정

```
1. Read → JSP 내 script 또는 외부 JS 파일
2. 이벤트 바인딩 → Ajax 호출 → 콜백 순으로 트레이스
3. 테스트시트 작성 (URL, 파라미터, 기대 동작)
4. Edit → 해당 함수/콜백만 수정
5. 동일 이벤트 중복 바인딩 여부 확인
```

### Java (Controller/Service/DAO)

```
1. Read → 문제 메서드
2. 호출 체인 추적: Controller → Service → DAO → SQL
3. 테스트시트 작성 (입력 파라미터 → 기대 출력)
4. Edit → 문제 메서드 내부만 수정
5. 인터페이스-구현체 일치 확인 (Service ↔ ServiceImpl)
```

### SQL (mapper.xml)

```
1. Read → DAO에서 호출하는 SQL id 확인
2. Read → mapper.xml에서 해당 SQL 문
3. 테스트시트 작성 (파라미터 → 기대 결과셋)
4. Edit → SQL 내부만 수정 (id, namespace 절대 미변경)
5. #{} vs ${} 확인 (SQL Injection 방지)
```

## 호출 예시

```
/tdd-fix board/list.jsp 목록이 안 나옴
/tdd-fix BoardController.java selectBoardList 빈 결과
/tdd-fix 게시판 상세 화면에서 첨부파일 다운로드가 안 돼
/tdd-fix login.jsp 비밀번호 입력란 엔터키 안 먹어
```

`$ARGUMENTS`에 파일명과 증상을 자유 형식으로 전달한다.
파일명이 없으면 Phase 0에서 사용자에게 확인한다.
