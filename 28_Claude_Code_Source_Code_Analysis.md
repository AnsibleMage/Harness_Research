# Claude Code 소스코드 유출 분석 — 내부 프롬프트에서 발견된 7가지 핵심 기법

## 개요

Claude Code의 소스코드(51만줄, TypeScript)가 유출되었다. 원본 레포지토리는 DMCA 요청으로 이미 내려갔고, 커뮤니티에 돌아다니던 복제본들도 대부분 정리된 상태다. 현재 남아 있는 것은 재작성이나 미러 성격의 레포 정도다.

**핵심 발견**: 유출된 코드 자체보다, Anthropic이 자사 AI에게 **어떻게 말하고 있었느냐**가 진짜 가치다. CLAUDE.md나 프롬프트에 오늘 당장 적용할 수 있는 기법들이 숨어 있었다.

**이 문서의 구성**: Anthropic이 Claude Code 내부 프롬프트에서 실제로 사용하고 있던 핵심 기법 7가지를, 쉬운 것부터 난이도 순서대로 정리한다. 앞쪽은 오늘 바로 쓸 수 있는 것들이고, 뒤로 갈수록 AI 활용 수준을 한 단계 올려줄 구조적인 이야기다.

---

## §0. 전제 — Claude Code는 챗봇이 아니라 에이전트 런타임이다

소스코드를 분석하면서 가장 먼저 눈에 들어온 것은 개별 기법이 아니라, Anthropic이 Claude Code를 **어떤 관점으로 만들었느냐**였다.

### 아키텍처 구조

소스코드의 핵심 루프:

```
사용자 입력 → AI가 도구 선택 판단 → 권한 확인 → 도구 실행 → 결과를 AI에게 반환 → 반복
```

이 루프가 **더 이상 도구를 쓸 필요가 없을 때까지** 반복된다.

### 챗봇 vs 에이전트

| 구분 | 일반 AI 챗봇 | Claude Code (에이전트) |
|------|-------------|---------------------|
| **비유** | 전화 상담원 | 현장에 직접 와서 일하는 엔지니어 |
| **행동** | 조언만 제공 | 실제로 장비를 만지고 문제를 고침 |
| **질문 방식** | "이거 어떻게 해?" | "먼저 이걸 조사하고, 그다음에 계획을 세우고, 실행하고, 검증해라" |
| **프롬프트** | 하나의 거대한 프롬프트 | 작은 단위의 작업으로 쪼개서 요청 |

이 관점이 중요한 이유: 에이전트한테 일을 시키는 방식과 챗봇한테 질문하는 방식은 완전히 다르다. 오늘 다루는 7가지 기법도 전부 이 **에이전트적 사고 방식** 위에서 작동한다.

## §1. 기법 1 — Do & Don't 대조 패턴 (난이도: ★☆☆)

### 문제

"간결하게 써 줘"라고 지시하면 Claude 나름대로는 간결하게 쓴 것인데, 사용자 기준에는 여전히 길 수 있다. "간결하게"의 기준이 사람마다 다르기 때문이다.

### Anthropic의 해법

소스코드에 이런 지시가 박혀 있었다:

> **"비슷한 코드 세 줄을 복붙하는 게 섣부른 추상화보다 낫다"**

추상적인 원칙이 아니라 **구체적인 판단 기준**을 준 것이다. "세 줄"이라는 숫자가 있으니까 AI가 어디서 멈춰야 할지 판단할 수 있다.

### 핵심: "하지 마라"를 같이 적는다

MagicDocs(문서화 도구) 프롬프트 예시:

| 문서화할 것 (Do) | 문서화하지 말 것 (Don't) |
|-----------------|----------------------|
| 코드만 봐서는 알 수 없는 패턴이나 주의사항 | 코드를 읽으면 바로 알 수 있는 것 |

뭘 해야 하는지만 쓰면 범위가 넓어서 AI가 과잉 생산한다. 뭘 하지 말아야 하는지를 같이 적으면 범위가 좁혀진다.

### 도구 선택에도 동일 패턴 적용

소스코드에 명시된 도구 선택 규칙:

| Do (사용할 것) | Don't (사용하지 말 것) |
|--------------|---------------------|
| 파일 검색: `Glob` 사용 | `find`나 `ls` 쓰지 마라 |
| 콘텐츠 검색: `Grep` 사용 | `grep`이나 `rg` 쓰지 마라 |

선호하는 도구와 금지하는 도구가 **한 줄에 나란히** 있다.

### 숫자를 넣으면 효과가 더 올라간다

소스코드에서 발견된 수치 기반 지시:

| 지시 | 수치 |
|-----|------|
| 도구 호출 사이의 텍스트 | **25단어** 이내 |
| 최종 응답 | **100단어** 이내 |

"짧게 써라"가 아니라 "25단어", "100단어"라는 **측정 가능한 기준**을 준 것이다.

### 실전 적용

| Before (비효과적) | After (효과적) |
|-----------------|---------------|
| "요약해 줘" | "세 문장으로 요약해 줘" |
| "자세히 설명해 줘" | "500자 내외로 설명해 줘" |
| "패키지 설치해 줘" | "pip을 사용할 것. conda를 쓰지 말 것" |

숫자가 들어가는 순간 AI가 멈출 지점이 생�다. Do & Don't가 결합되면 AI의 행동 범위가 정밀하게 좁혀진다.

## §2. 기법 2 — 결과 검증 강제 패턴 (난이도: ★☆☆)

### 문제

"모든 테스트가 통과했습니다"라고 AI가 말하는데 실제로 확인해 보면 에러가 있는 경우. AI는 기본적으로 자신감 있게 대답하려는 경향이 있어서, "모르겠다"고 말하는 것보다 "했습니다"라고 말하는 쪽으로 기울어져 있다.

### Anthropic의 해법

소스코드에 박힌 지시:

> **"작업 완료를 보고하기 전에 실제로 작동하는지 검증해라. 테스트를 돌려라. 스크립트를 실행해라. 출력을 확인해라. 검증할 수 없으면 검증할 수 없다고 명시적으로 말해라."**

핵심은 마지막 문장이다. "검증할 수 없으면 검증할 수 없다고 말해라" — 명시적으로 **모른다고 말해도 된다는 허가**를 줘야 한다.

### 금지 지시의 구체성

소스코드에서 발견된 금지 목록:

| # | 금지 행동 |
|---|----------|
| 1 | 테스트가 통과했다고 **거짓 주장**하지 마라 |
| 2 | 실패한 체크를 숨기거나 단순화해서 **성공처럼 포장**하지 마라 |
| 3 | 미완성된 작업을 **완료라고 표현**하지 마라 |
| 4 | 결과를 **조작하거나 예측**하지 마라 |
| 5 | — 산문 형태로도 하지 마라 |
| 6 | — 요약 형태로도 하지 마라 |
| 7 | — 구조화된 출력 형태로도 하지 마라 |

"하지 마라"만으로는 부족하다. **나쁜 행동이 나타날 수 있는 형태까지 열거**해야 효과적이다.

### 비유: 아이에게 "과자 먹지 마"

"과자 먹지 마"라고만 하면 아이는 "초콜릿은 과자가 아니다"라고 주장하면서 먹을 수 있다. "과자, 초콜릿, 젤리, 아이스크림 전부 먹지 마"라고 해야 빠져나갈 구멍이 없어진다.

### 금지에는 반드시 대안을 제시

> 결과를 **조작하지 말고**, 대신 **현재 상태를 보고**해라.

금지만 하면 AI가 뭘 해야 할지 모른다. **"이건 하지 마 → 대신 이걸 해라"**는 쌍으로 적어야 한다.

### 실전 적용

```
확인할 수 없는 사항은 추측하지 말고 "확인 불가"라고 명시할 것.
추측, 가정, 유추 형태 모두 포함.
```

## §3. 기법 3 — 3중 반복 패턴 (난이도: ★★☆)

### 발견

소스코드에서 가장 인상적이었던 발견 중 하나. Claude Code의 자동 압축 기능에서 **"도구를 호출하지 말라"는 지시가 프롬프트 안에 세 번 등장**한다.

| 위치 | 내용 |
|------|------|
| **맨 앞** | CRITICAL: 텍스트로만 응답하라. 도구를 호출하지 마라 |
| **중간** | 지시사항 본문 안에 한 번 더 |
| **맨 끝** | 리마인더: 도구를 호출하지 마라 |

### 왜 세 번인가 — 최근 편향(Recency Bias)

AI 모델에는 **최근 편향**이라는 특성이 있다. 가장 최근에 읽은 정보에 더 많이 영향을 받는다. 사람도 긴 문서를 읽으면 마지막에 읽은 내용이 가장 기억에 남는 것과 같다.

| 배치 위치 | 효과 |
|----------|------|
| **맨 앞** | 첫 읽기에서 주의를 잡음 |
| **중간** | 본문 맥락에서 강화 |
| **맨 끝** | 실행 직전에 한 번 더 상기 |

연구에 따르면 단일 강한 문장 하나보다 **세 군데에 분산 배치하는 것이 준수율이 높다**.

### 실전 적용

정말 중요한 규칙이 있으면:

1. CLAUDE.md **맨 위**에 한 번
2. **관련 섹션**에 한 번
3. **맨 아래**에 리마인더로 한 번

```
예: "이 프로젝트에서 데이터베이스를 직접 삭제하는 명령은 절대 실행하지 마라"
```

이런 규칙은 세 번 적어 놔도 과하지 않다. 앞에서 다뤘던 Do & Don't 형식으로 각 위치에 적으면, 위치가 다르니까 같은 말을 반복해도 자연스럽다.

## §4. 기법 4 — 턴 예산 전략과 컨텍스트 비용 인식 (난이도: ★★☆)

### 발견

소스코드의 메모리 추출 에이전트에 이런 지시가 있었다:

> **"턴 1에서 업데이트할 모든 파일의 Read 호출을 병렬로 발행해라. 턴 2에서 모든 Write와 Edit 호출을 병렬로 발행해라. 읽기와 쓰기를 여러 턴에 걸쳐 섞지 마라."**

### 왜 이렇게 하는가 — 마트 장보기 비유

파일을 하나 읽고 → 하나 수정하고 → 또 하나 읽고 → 또 하나 수정하면 매번 API 호출이 왕복한다. 마트에서 우유를 사러 갔다가 돌아오고, 다시 빵을 사러 갔다가 돌아오고, 다시 계란을 사러 가는 것과 같다. **한 번에 다 사오는 게 효율적**이다.

### 병렬 실행 구조

소스코드에서 확인된 실행 방식:

| 도구 유형 | 실행 방식 | 동시 실행 수 |
|----------|----------|------------|
| **안전한 도구** (파일 읽기, 텍스트 검색, 파일 목록 조회) | **병렬** | 최대 10개 동시 |
| **위험한 도구** (파일 수정, 명령어 실행) | **순차** | 하나씩 |

### 컨텍스트 = 비용 — 택시 미터기 비유

이 턴 예산은 단순히 속도 문제만이 아니라 **돈 문제**이기도 하다.

- 컨텍스트에 올라간 모든 파일이 **매 턴마다 API로 전송**된다
- 파일 하나를 컨텍스트에 올리면 그 파일의 토큰이 **매 API 호출마다 과금**된다
- 파일을 열어보고 안 쓰더라도 과금된다

**비유**: 택시 미터기가 계속 돌아가는데, 짐을 트렁크에 실을수록 미터기가 더 빨리 돌아가는 것과 같다.

### 비용 구조

Claude Code $100 플랜에는 실제로 **$1,000 이상의 API 보조금**이 포함되어 있는 것으로 알려져 있다. 관리 없이 사용하면 이 보조금이 생각보다 빠르게 소진된다. **컨텍스트 관리가 곧 비용 관리**라는 인식 자체가 중요하다.

### 실전 적용

1. 여러 파일 수정 시: **"먼저 관련 파일 전부를 읽어보고 분석한 다음에 수정해라"**고 순서를 명시
2. 대화가 길어지면: `/compact` 명령으로 주기적으로 압축
3. 프롬프트의 **작업 순서를 명시**하는 것만으로 토큰 절약과 속도 향상이 동시에 달성

## §5. 기법 5 — Chain of Thought 스트리핑 패턴 (난이도: ★★★)

### 배경

Chain of Thought(단계별 사고) — AI에게 "먼저 생각을 정리한 다음에 답변해라"고 시키면 답변 품질이 올라간다. 이건 이미 잘 알려진 사실이다.

### Anthropic의 실전 적용 — 생각은 하게 하되, 출력에서 잘라낸다

소스코드의 자동 압축 프롬프트에서 발견된 구조:

1. **"분석 내용을 `<analysis>` 태그 안에 작성해라"**라는 지시
2. 코드에서 이 **`<analysis>` 태그를 통째로 제거**하는 로직

무슨 뜻이냐면: **생각은 하게 하되, 생각한 내용은 출력에서 잘라내는 것**이다.

### 비유: 시험 답안지

시험 답안지에 연습장 풀이 과정을 적게 하고, 답만 깔끔하게 옮겨 적게 한 다음, **연습장은 수거**하는 것과 같다. 풀이 과정 덕분에 답의 정확도는 올라가지만, 최종 결과물은 깔끔하다.

### 왜 중요한가 — 두 가지 차원

| 차원 | 내용 |
|------|------|
| **1차원** (잘 알려짐) | 생각하게 만들면 품질이 올라간다 |
| **2차원** (소스코드 발견) | 그 생각 과정을 **어떻게 관리하느냐** |

생각 과정이 출력에 그대로 포함되면 두 가지 문제가 생긴다:
1. **토큰을 많이 소모**한다
2. **사용자에게 불필요한 정보가 노출**된다

Anthropic은 **"태그 삽입 후 제거"**라는 구조로 깔끔하게 해결했다.

### 실전 적용

```
먼저 분석 내용을 <thinking> 태그 안에 적어라.
최종 출력에는 <thinking> 태그를 제외하고 결론만 작성해라.
```

- **API 직접 사용 시**: 후처리에서 태그를 제거하면 완벽하게 작동
- **대화형 인터페이스**: AI에게 "태그 안의 내용은 출력하지 마라"고 지시하는 방식으로 유사 적용 가능

**핵심**: 생각하게 만드는 것과 생각을 관리하는 것은 **다른 차원의 기술**이다. Anthropic이 자사 제품에서 이 두 번째 차원을 이미 실전 적용하고 있었다.

## §6. 기법 6 — Skills Description 예산 (난이도: ★★★)

### 발견: 1% 예산 제한

소스코드에서 발견된 핵심 수치:

| 파라미터 | 값 | 의미 |
|---------|-----|------|
| `skillBudgetContextPercent` | **0.01** | 전체 컨텍스트 윈도우의 **딱 1%** |
| 개별 스킬 description 상한 | **250자** | 트리거 판단의 유일한 근거 |

### 1%의 의미

- 최근 Claude 모델 기본 컨텍스트 윈도우: **100만 토큰**
- 스킬즈 리스팅에 쓸 수 있는 예산: **1만 토큰**
- 한 스킬의 description이 250자니까 이론상 여유가 있어 보이지만, 실제로는 각 스킬의 메타데이터와 포매팅 오버헤드가 있어서 생각보다 빠르게 차오른다

### 예산 초과 시 — 계급 구조

예산을 초과하면 다음 순서로 축소된다:

| 단계 | 동작 |
|------|------|
| 1단계 | 사용자가 만든 **커스텀 스킬의 description부터 축소** |
| 2단계 | 그래도 부족하면 커스텀 스킬은 **이름만 표시** |
| (보호) | Anthropic의 **공식 번들 스킬은 항상 전체 description 유지** |

### 비유: 비행기 좌석

좌석이 부족한 비행기에서 **퍼스트 클래스 승객은 무조건 탑승**하고, **이코노미 승객부터 짐을 줄이거나 탑승 거부**당하는 구조. 사용자가 만든 스킬이 이코노미 좌석이다.

### 실전 영향

- 스킬을 10개, 20개 만들어 놓으면 **뒷번호 스킬은 description이 잘리거나 이름만 표시**
- Description이 잘린 스킬은 Claude가 **언제 써야 하는지 판단할 근거가 사라진다** — 트리거가 죽는 것
- 따라서:
  1. Description을 **250자 안에 정말 압축적으로** 쓸 것
  2. 스킬 개수를 무작정 늘리기보다 **정말 자주 쓰는 핵심 스킬 위주로 관리**할 것

## §7. 기법 7 — 컨텍스트 오염 방지 (난이도: ★★★)

### 발견

Claude Code의 Fork Agent 시스템에서 발견된 규칙. Fork는 대화 중간에 **분신을 만들어서 별도 작업을 시키는 기능**인데, 이 분신이 작업하는 동안 **본체가 분신의 작업 로그를 들여다보는 것을 명시적으로 금지**하고 있었다.

소스코드 원문:

> **"사용자가 명시적으로 요청하지 않는 한 출력 파일을 Read하거나 Tail하지 마라. 작업 중간에 트랜스크립트를 읽으면 포크의 도구 노이즈가 본 컨텍스트로 유입되고, 이는 포크를 쓰는 목적 자체를 무산시킨다."**

"목적 자체를 무산시킨다"라는 표현이 인상적이다. 단순히 "하지 마라"가 아니라 **왜 하면 안 되는지를 설명**하고 있다.

### 비유: 요리사 두 명

요리사 두 명이 따로 요리하고 있는데, 한 명이 다른 요리사의 작업대를 계속 들여다보면 자기 요리에 다른 재료가 튀어 들어가는 것과 같다.

### 적용 범위

이 원리는 Claude Code에서 서브에이전트를 쓰거나 백그라운드 태스크를 돌릴 때 동일하게 적용된다. 작업을 분리해 놓은 이유가 있는데, 중간에 결과를 당겨 와서 현재 대화에 집어넣으면 **맥락이 오염**된다.

### 실전 적용

CLAUDE.md에 이렇게 적용할 수 있다:

```
프로젝트에서 서로 다른 모듈을 동시에 작업할 때,
A 모듈의 에러 로그를 B 모듈 작업 중에 참조하지 말 것.
```

### §4와의 관계

| 기법 | 설계 대상 |
|------|----------|
| §4 턴 예산 전략 | **작업 순서**를 설계하는 것 |
| §7 컨텍스트 오염 방지 | **작업 경계**를 설계하는 것 |

순서와 경계, 이 두 가지를 함께 관리해야 Claude Code가 복잡한 프로젝트에서도 흔들리지 않는다.

## §8. 시사점 — 엔지니어링 패러다임의 전환

### Anthropic이 자사 AI에게 한 말

소스코드 내부 프롬프트 중 발견된 구절:

> **"당신은 단순 실행자가 아니라 협업자다. 사용자는 당신의 순종이 아니라 판단력을 필요로 한다."**

Anthropic이 자기 AI한테 이렇게 말하고 있다는 사실 자체가 시사적이다.

### 산업 변화

- 이미 일부 기업에서는 엔지니어에게 **스펙 파일과 테스트만 작성**하게 하고, 실제 코딩은 AI에게 맡기는 방식을 도입
- 월 **$2,000~$3,000 수준의 AI 크레딧**을 엔지니어에게 지급하는 회사도 등장
- 코드를 잘 짜는 능력보다 **무엇을 만들어야 하는지 정의하는 능력**이 더 가치 있어지는 흐름

### 핵심 역량의 전환

테스트를 먼저 작성하고 AI에게 구현을 맡기는 워크플로우에서는:

| 핵심 역량 | 설명 |
|----------|------|
| **시스템 설계** | 무엇을 만들지 정의 |
| **맥락 관리** | AI에게 올바른 컨텍스트 제공 |
| **작업 분해** | 작업을 잘게 쪼개는 능력 |

### 7가지 기법의 본질

| # | 기법 | 본질 |
|---|------|------|
| 1 | Do & Don't 대조 | AI의 행동 범위를 정밀하게 좁히기 |
| 2 | 결과 검증 강제 | 검증 게이트 만들기 |
| 3 | 3중 반복 | 중요한 규칙을 반복 배치하기 |
| 4 | 턴 예산 전략 | 작업 순서를 설계하기 |
| 5 | CoT 스트리핑 | AI의 사고 과정을 구조화하기 |
| 6 | Skills Description 예산 | 시스템의 자원 제약을 이해하기 |
| 7 | 컨텍스트 오염 방지 | 작업 경계를 설계하기 |

이건 전부 **AI한테 좋은 맥락을 제공하는 방법**이다. 코드를 짜는 기술이 아니라, **코드를 짜게 만드는 기술**이다.

### 결론

소스코드가 공개됐다고 모두가 같은 출발선에 서는 게 아니다. 코드는 인프라일 뿐이고, 차별화는 그 인프라 위에서 **자기만의 워크플로우와 맥락을 얼마나 잘 구축하느냐**에 달려 있다.

- 스킬즈를 만들고
- CLAUDE.md를 가꾸고
- 자신의 도메인에 맞는 판단 기준을 기록하는 것

결국 여기서 차이가 벌어진다. 도메인 지식, 업무 판단 기준, 좋은 결과물을 골라내는 눈 — 이것들은 소스코드가 공개되든 안 되든 바뀌지 않는다.
