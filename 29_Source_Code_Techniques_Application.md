## 개요

[[28_Claude_Code_Source_Code_Analysis.md|28번 문서]]에서 정리한 Claude Code 소스코드 내부 프롬프트 7가지 기법을, 현재 V5.2.0-WE 시스템(`CLAUDE.md` + `rules/*.md`)과 대조 분석하고, **어디에 어떤 문구를 추가해야 하는지** 구체적으로 정리한다.

### 적용도 판정 기준 (v1.1.0 추가)

> 시니어 검증 지적: "부분 적용"의 기준이 모호. 구체적 채점 기준을 명시하라.

| 판정 | 기준 | 표기 |
|------|------|------|
| **적용됨** | 기법의 핵심 요소 중 **80%+** 구현 | 🟢 (3/3, 4/5 등) |
| **부분 적용** | 핵심 요소 중 **30~79%** 구현 | 🟡 (1/3, 2/5 등) |
| **미적용** | 핵심 요소 중 **30% 미만** 구현 | 🔴 (0/3, 0/4 등) |

### 격차 요약 매트릭스

## §1. 기법 1 적용 — Do & Don't 대조 패턴

### 현재 상태 (적용도: 🟡 2/4)

| 핵심 요소 | 현재 | 비고 |
|----------|------|------|
| ① 도구 선택 Do/Don't | ✅ 적용 | 시스템 프롬프트 내장 (Glob↔find, Grep↔grep) |
| ② 행동 규칙 Do/Don't 쌍 | ⚠️ 일부 | 축약 금지(§2.4), L1 #3에 Do/Don't 존재. 그러나 대부분 규칙이 Do만 |
| ③ 수치 기준 | ❌ 없음 | "간결" 원칙만. 숫자 기준 전무 |
| ④ 문서 작성 Do/Don't | ❌ 없음 | 템플릿 규칙만. 쓰지 말아야 할 것 명시 없음 |

### 격차

기존 축약 금지(§2.4)와 L1 #3에 Do/Don't 형태가 부분적으로 존재하지만, **수치 기준이 전무**하고 **체계적 Do/Don't 쌍 적용이 안 되어 있다**.

### 적용 위치 및 추가할 문구

#### 위치 1: `rules/behavioral-standards.md` (신규 파일) — Output Standards 섹션

```markdown
## 출력 기준 (Do & Don't)

| Do | Don't | 수치 가이드 |
|----|-------|-----------|
| 도구 호출 사이 설명은 간결히 한다 | 도구 호출 사이에 분석 과정을 장황하게 풀지 않는다 | ~25단어 권장 |
| 최종 응답은 핵심만 전달한다 | 요약, 반복, 불필요한 전이구("자 그러면", "좋습니다")를 쓰지 않는다 | Effort별 분화 (아래 참조) |
| 코드 수정 시 변경점만 설명한다 | 변경하지 않은 코드를 다시 설명하지 않는다 | 변경 파일당 ~3줄 권장 |
| 비슷한 코드 3줄 복붙을 허용한다 | 1회성 로직에 불필요한 헬퍼/유틸 함수를 만들지 않는다 | 반복 3줄 이내면 추상화 불요 |

### Effort Level 연동 수치 가이드

| Effort | 최종 응답 길이 가이드 | 적용 |
|--------|--------------------|----|
| **LOW** (HotfixChain) | ~100단어. 진단→수정→결과만 (잔여 보고/에러 보고는 예외) | 엄격 적용 |
| **MEDIUM** (DevChain 등) | 품질 우선, 필요한 만큼 작성 | CLEAR "Concise" 판단 |
| **HIGH** (ResearchChain 등) | 품질 우선, 필요한 만큼 작성. 심층 분석 우선 | CLEAR "Concise" 판단 |
| **N/A** (Simple Task) | 수치 가이드 미적용. CLEAR "Concise" 자연 적용 | 체인 생략 시 |

> ⚠️ 수치는 **가이드라인**이지 경직된 제한이 아니다. 필요한 설명을 잘라내서는 안 된다.
```

#### 위치 2: `rules/orchestration.md` §2.4 — 임의 축약 금지 규칙 확장

현재 문구를 Do/Don't 테이블로 변환:

```markdown
> ⚠️ 체인 선택 후, 정의된 모든 에이전트를 순서대로 실행한다.

|----|-------|

| 체인 축소가 필요하면 앤에게 확인한다 | 아리가 독자적으로 에이전트 수를 줄이지 않는다 |
```

## §2. 기법 2 적용 — 결과 검증 강제 패턴

### 현재 상태 (적용도: 🟡 3/5)

| 핵심 요소 | 현재 | 비고 |
|----------|------|------|
| ① 검증 루프 | ✅ 적용 | Boris Tip #5, MAX 3회, 6개 체인 |
| ② 3종 리뷰어 | ✅ 적용 | logic ∥ security ∥ edge-case 병렬 |
| ③ Stop Hook | ✅ 적용 | debug-residue-check.sh |
| ④ "확인 불가" 허가 | ❌ 없음 | AI가 "모르겠다"고 말해도 된다는 명시 없음 |
| ⑤ 금지 형태 열거 | ❌ 없음 | 거짓 보고의 구체적 형태 열거 없음 |

### 격차

검증 **구조**(루프, 리뷰어, Hook)는 우수하지만, 검증 **정직성**(거짓 보고 금지, 확인 불가 허용)에 대한 명시가 없다.

### 적용 위치 및 추가할 문구

#### 위치 1: `rules/behavioral-standards.md` — Verification Honesty 섹션

```markdown
## 검증 정직성

작업 완료 보고 전 반드시 실제 검증을 수행한다. 검증할 수 없으면 "검증 불가"라고 명시한다.

| 금지 행동 | 대안 행동 |
|----------|----------|
| 테스트가 통과했다고 거짓 주장 | 실제 Bash 실행 결과를 첨부 |
| 실패를 숨기거나 성공처럼 포장 | 실패 내용을 그대로 보고 |
| 미완성 작업을 "완료"라고 표현 | 완료된 부분과 잔여 부분을 분리 보고 |
| 결과를 추측·가정·유추로 생성 | "확인 불가 — [이유]"로 명시 |
| — 산문·요약·구조화 출력 모두 포함 | 형태 무관하게 사실만 보고 |

> ⚠️ "모르겠다", "확인 불가"는 허용된 응답이다. 거짓 확신보다 정직한 불확실성이 낫다.
```

#### 위치 2: `rules/orchestration.md` §2.4 — 검증 루프 프로토콜 하단에 추가

```markdown
#### 검증 정직성 규칙 (Verification Honesty)

검증 루프 내에서도 동일 적용:
- 리뷰어 결과를 축소·생략하여 "통과"로 보고하지 않는다 → Critical은 건수·내용을 정확히 전달한다
- Bash 테스트 실패 시 "부분 통과"로 포장하지 않는다 → 실패 로그 원문을 첨부한다
- MAX 3회 소진 후에도 "거의 해결"이라 하지 않는다 → 잔여 이슈를 목록으로 명시한다
```

### 3중 반복 대상 선정 (v1.1.0 — 복합 지표 적용)

> 시니어 검증 Warning #4: "위반 빈도 × 피해 크기" 복합 지표로 변경하라.

| 규칙 후보 | 위반 빈도 | 피해 크기 | 복합 점수 | 선정 |
|----------|----------|----------|----------|------|
| **체인 축약 금지** | 중 (AI의 효율 판단 편향) | 높 (미검증 코드 배포) | **높** | ✅ 선정 |
| **비위맞추기 금지** | 높 (AI의 동의 편향) | 높 (의사결정 품질 하락) | **높** | ✅ 선정 |
| **Teammate 메모리 저장 금지** | 낮 (명확한 기계적 규칙) | 높 (데이터 정합성 훼손) | **중** | ✅ 선정 |
| 1 Prompt = 1 Memory | 중 (생략 유혹) | 중 (맥락 전달 단절) | **중** | ❌ 탈락 |

**"1P=1M" 탈락 사유**: 위반 시 피해가 "맥락 전달 단절"이지만, 이는 현 세션 내에서 즉시 감지·보정 가능하다(앤이 "메모리 저장했어?"로 확인). 반면 체인 축약이나 비위맞추기는 앤이 감지하기 어렵고 발견 시 이미 피해가 발생한 상태다. **감지 난이도**를 고려하면 상위 3개가 더 급하다.

```markdown
|----|-------|
| 여러 파일 수정 시 **Turn 1에서 관련 파일 전부를 Read**한다 (병렬) | 한 파일 Read → Edit → 다음 파일 Read → Edit를 반복하지 않는다 |
| **Turn 2에서 모든 Edit/Write를 병렬 발행**한다 | 읽기와 쓰기를 여러 턴에 걸쳐 섞지 않는다 |
| 안전한 도구(Read, Glob, Grep)는 병렬 실행한다 | 위험한 도구(Bash, Edit)를 무분별하게 병렬 발행하지 않는다 |

> **컨텍스트 = 비용**: 컨텍스트에 올라간 모든 파일이 매 턴마다 API로 전송된다. 불필요한 파일을 컨텍스트에 올리지 않는다. 대화가 길어지면 `/compact`로 압축한다.
> **검증 루프 예외**: 검증 루프 내 반복(수정→리뷰→수정)에서는 턴 분리보다 검증 품질을 우선한다.
```

#### 위치 2: `rules/orchestration.md` §2.5 — Teammate 행동 규칙 7번으로 추가

```markdown
7. **턴 예산 준수** — Read 병렬 발행 → 분석 → Write 병렬 발행. 읽기와 쓰기를 턴 단위로 분리
```

## §5. 기법 5 적용 — Chain of Thought 스트리핑 패턴

### 현재 상태 (적용도: 🟡 2/3 — v1.1.0 재판정)

> 시니어 검증 Warning #5: "자체 해결"이 아닌 "부분 적용"으로 재판정하라.

| 핵심 요소 | 현재 | 비고 |
|----------|------|------|
| ① 내부 사고 관리 | ✅ 적용 | Extended Thinking (모델 내장) |
| ② Thinking Process 명시 | ✅ 적용 | CLAUDE.md §1 (인식→탐색∥리스크→선택→검증) |
| ③ 에이전트 간 출력 정제 | ❌ 없음 | Teammate→Lead 메시지에 분석 과정 포함 가능 |

**재판정 사유**: Extended thinking은 **모델의 내부 사고**를 관리하는 것이지, **에이전트 간 통신의 출력 정제**와는 다른 차원이다. Teammate가 Lead에게 보내는 메시지에 도구 호출 로그, 탐색 이력, 중간 분석이 포함될 수 있으며, 이를 관리하는 규칙이 없다.

### 적용 위치 및 추가할 문구

#### 위치: `rules/orchestration.md` §2.5 — Teammate 행동 규칙 8번으로 추가

```markdown
8. **출력 정제** — Teammate는 Lead에게 결과를 전달할 때 아래 형식을 준수한다:
   - **포함할 것**: 결론, 핵심 근거(~3개), 발견한 문제점, 권고사항
   - **포함하지 말 것**: 도구 호출 로그, 파일 탐색 이력, 시행착오 기록
   - 분석 깊이가 필요하면 결과 파일(Write)로 분리하고 Lead에게는 요약만 전달
   - **리뷰어 예외**: logic/security/edge-case/quality 리뷰어가 Teammate로 실행될 때는 분석 근거를 포함한다. 검증 루프의 code_developer가 수정 근거로 사용하기 때문이다.
```

#### CLAUDE.md 직접 반영은 하지 않는 이유

| 이유 | 설명 |
|------|------|
| 모델 내장 | Extended thinking이 메인 세션의 CoT 스트리핑을 수행 |
| API 레벨 기능 | `<analysis>` 태그 삽입+제거는 코드 레벨 후처리 — CLAUDE.md 지시로 대체 불가 |
| rules/로 커버 | Teammate 출력 정제를 orchestration.md에서 관리 |

### 격차

**가장 큰 격차**. 43개+ 스킬은 1% 예산(약 1만 토큰)을 소진시킬 수 있다. 뒷번호 커스텀 스킬의 description이 잘리면 트리거가 죽는다.

> 시니어 검증 Warning #6: "가장 큰 격차"라면서 Phase 3은 모순. description 감사는 Phase 1로 승격하라.

### 적용 방안 — 3단계

#### 방안 A: 스킬 description 감사 (Phase 1 — 즉시 실행)

전체 스킬의 description을 점검하고 250자 이내로 압축한다.

**감사 기준:**

| 항목 | 기준 |
|------|------|
| 글자 수 | **250자** 초과 시 압축 필수 |
| 트리거 키워드 | description 첫 줄에 핵심 트리거 키워드 배치 |
| 중복 제거 | 유사 스킬 간 description 차별화 |
| 한국어/영어 | 트리거 키워드는 **한국어 + 영어** 병기 |

**감사 실행 방법:**
```bash
# 모든 커스텀 스킬의 SKILL.md에서 description 추출
# → 글자 수 측정 → 250자 초과 목록 작성 → 압축
```

#### 방안 B: 스킬 티어링 (Phase 3)

스킬을 사용 빈도 기반으로 3개 티어로 분류한다.

| 티어 | 기준 | 관리 |

| **Tier 2 (보조)** | 월 2~3회 사용 | description 압축, 조건부 활성 |
| **Tier 3 (저빈도)** | 월 1회 이하 | 필요 시 수동 호출 (`/스킬명`), description 최소화 |

#### 방안 C: CLAUDE.md에 인식 규칙 추가 (Phase 3)
